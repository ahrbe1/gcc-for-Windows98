## 1. What this repo is

A reproducible Docker-based build that produces a working GCC 11.1.0 toolchain for Windows 98. The upstream feat is from [fsb4000/gcc-for-Windows98](https://github.com/fsb4000/gcc-for-Windows98); this repo wraps the patch recipe in a fully scripted, pinned, containerized build.

It produces **three** archive artifacts plus per-archive manifests and a Docker image:

| Artifact            | Path under `repro/out/package/`                | Host    | Target               |
| ------------------- | ---------------------------------------------- | ------- | -------------------- |
| Cross toolchain     | `gcc-win98-cross-toolchain.tar.xz` + `.json`         | Linux   | `i686-w64-mingw32`   |
| Native toolset      | `gcc-win98-native-toolchain.zip` + `.json`       | Win98   | `i686-w64-mingw32`   |
| Extras toolset      | `gcc-win98-native-toolchain-extras.zip` + `.json`               | Win98   | (user tools)         |
| Consumer image      | `gcc-win98-consumer:latest`                    | â€”       | â€”                    |

The two Win98-hosted archives ship as `.zip` (not `.tar.xz`) so 7zip 9.20 on Win98 SE can extract them in one pass without spilling a ~700 MB scratch tar. zip also has no hardlink/symlink concept, which sidesteps a FAT32 trap: tar archives hardlinks (e.g. `g++.exe â†’ c++.exe`, `ld.exe â†’ ld.bfd.exe`) as metadata-only entries, and FAT32 can't represent them â€” they materialize as 0-byte stubs and Win98 reports "Error in EXE file" at load time. The cross toolchain stays `.tar.xz` because it's only ever consumed inside the Linux consumer container, where xz is fast.

The **extras archive** bundles Win98-hosted user tools: `busybox.exe`/`sh.exe`, `make.exe`, `ctags.exe`, `diff.exe`/`cmp.exe`, `patch.exe`, `gdb.exe`, `muon.exe`, `jq.exe`, `tcc.exe` (TinyCC â€” a second, much smaller C compiler running on-host; see Â§5.11), ~112 byte-identical `bb-shim.exe` copies under per-applet names (`ls.exe`, `wget.exe`, `bzip2.exe`, ...) for FAT32-symlink-less dispatch into busybox (see Â§5.10), plus a tiny `bcrypt.dll` shim (see Â§5.7) that satisfies `gdb.exe`'s libstdc++-via-`std::random_device` import, and two Win9x diagnostics (`consdiag.exe` for stdio/console probes, `sockdiag.exe` for Winsock socket-creation probes â€” see Â§8 file map). It's gated behind `BUILD_EXTRAS=1` (default on locally, on for tag pushes in CI, off for branch CI runs) so iterating on the cross/native phases doesn't pay the extras cost every time.

GitHub Actions ([.github/workflows/docker.yml](.github/workflows/docker.yml)) is **manual-dispatch only** â€” it runs `repro/build.sh` when invoked from the Actions UI and uploads `repro/out/package/*` as workflow artifacts. Tag pushes additionally attach those archives to a GitHub release. No Docker Hub publishing. **The GitHub release is the user's release surface** â€” shipped archives need to land in `repro/out/package/` as either `.tar.xz` (cross) or `.zip` (native/extras).

## 2. The Win98 constraints you must respect

These constraints are the entire reason this project exists. Anything you add must obey them, or it won't run on Win98 â€” even if it compiles.

| Constraint              | Why                                                                                       |
| ----------------------- | ----------------------------------------------------------------------------------------- |
| CRT = `msvcrt.dll`      | UCRT requires Windows 10+. mingw-w64 is configured with `--with-default-msvcrt=msvcrt`.   |
| Threading = `pthread9x` | `_GLIBCXX_THREAD_ATEXIT_WIN32` requires Vista+. pthread9x is `JHRobotics/pthread9x`.       |
| `MajorOSVersion â‰¤ 4`    | Enforced by [`scripts/verifiers/pe-win98-check.sh`](repro/scripts/verifiers/pe-win98-check.sh) on every produced `.exe`. |
| `MajorSubsystemVersion â‰¤ 4` | Same verifier; some toolchains write `4.0` for the OS field but `5.0` (the binutils default) for the subsystem field. Force with `-Wl,--major-subsystem-version=4` â€” already in `WIN98_TARGET_LDFLAGS`. |
| No forbidden imports    | Same verifier rejects `*ucrt*`, `api-ms-win-*`, `vcruntime*` plus any DLL not in the Win98 SE export snapshot at [`data/win98se-api-allowlist.json`](repro/data/win98se-api-allowlist.json), and any function not in that DLL's exports. |
| Static link C++         | `-static-libgcc -static-libstdc++` is the convention. Also required on `gdbserver.exe` â€” without it, the link picks up `libgcc_s_dw2-1.dll` and the binary is unloadable on Win98. |
| Win98-safe CPPFLAGS/LDFLAGS | Every native build script must propagate `$WIN98_TARGET_CPPFLAGS` (`-D_WIN32_WINNT=0x0400 -DWINVER=0x0400`) and `$WIN98_TARGET_LDFLAGS` (`-Wl,--disable-dynamicbase -Wl,--disable-nxcompat -Wl,--major-subsystem-version=4`) into the configure-time env, otherwise mingw-w64 feature-detects Vista+ paths and the PE gets DllCharacteristics Win98 doesn't recognize. |
| Target triple           | `i686-w64-mingw32` (32-bit). No 64-bit anywhere.                                          |

Any new tool's build must end with PE-checking every `.exe` and `.dll` it produces and failing the build if the check fails. The verifier is sourceable â€” see how [`verify-extras-package.sh`](repro/scripts/verifiers/verify-extras-package.sh) uses it.

**Bundled-DLL escape hatch.** When a binary needs an import the loader will satisfy from a DLL we ship in the same package (App Directory search), set `PE_CHECK_BUNDLED_DLLS="foo.dll bar.dll"` before invoking the verifier â€” those names skip both the system-allowlist and per-function checks. The bundled DLL itself still goes through the full check on its own merits. Current use: the `bcrypt.dll` shim alongside `gdb.exe` (see Â§5.7).

**Function-level escape hatch.** When the *DLL* exists on Win98 but a *function* the consumer wants doesn't (e.g. `kernel32!GetFinalPathNameByHandleA`, `ws2_32!getaddrinfo`), the compile-time API shim at [`repro/win98-compat/`](repro/win98-compat/) macro-redirects the call to a `GetProcAddress`-probing wrapper. The shim ships in the cross-toolchain sysroot â€” see Â§5.6.

**Behavioral denylist (stub-only exports).** Some functions are present in Win98 SE's export table â€” so they bind at PE load time â€” but are NT-era stubs that always return failure on the Win9x kernel. The allowlist accepts them (it reflects the actual export table verbatim, generated from a real Win98 SE install). A second-pass *denylist* at [`repro/data/win98-behavioral-denylist.json`](repro/data/win98-behavioral-denylist.json) names symbols to reject even though the allowlist accepts them; the verifier surfaces those as `Win98 stub-only (binds but non-functional): <dll>:<sym>` with `PE_CHECK_BAD_KIND=denied`. Currently seeded with the 98 NT-security exports in `advapi32.dll` (`Open*Token` / `*SecurityDescriptor` / `*Sid` / `*Acl` / `*SecurityInfo` / `Logon*` / `Lookup*` / `Impersonate*` / etc.). When you ship a binary that trips one of these, the fix is to patch out the call site at the source â€” convention is to wrap with `#ifdef _WIN98_PORT` short-circuits to the existing failure-path defaults (see busybox-w32 patch 0002 and make patch 0001 for the pattern), then add `-D_WIN98_PORT` to the tool's build script. Don't grow the denylist to whitelist a tool â€” add a tool patch instead. The denylist file's `description`/`rationale` fields explain when to add a symbol (it must be confirmed-present in `win98se-api-allowlist.json` AND confirmed-non-functional on the Win9x kernel).

## 3. Pipeline shape

### 3.1 Entry points

- [`repro/build.sh`](repro/build.sh) â€” one-shot host orchestrator. Builds the `toolchain-builder` image, runs the pipeline inside it, then builds the `consumer` image with the artifacts embedded, then runs smoke tests. Sources `repro/.env` so persisted settings (`JOBS`, `MATRIX`, `BUILD_EXTRAS`) affect orchestration; shell-supplied overrides win over `.env`.
- [`repro/scripts/run-toolchain-build.sh`](repro/scripts/run-toolchain-build.sh) â€” declares three arrays of build steps (`CROSS_STEPS`, `NATIVE_STEPS`, `EXTRAS_STEPS`) and runs each via `builder_script` (a `docker compose exec` wrapper from [`scripts/lib/common.sh`](repro/scripts/lib/common.sh)). **This is where you add new build phases.** Extras phase is gated on `BUILD_EXTRAS=1`.
- [`repro/scripts/run-smoke-pipeline.sh`](repro/scripts/run-smoke-pipeline.sh) â€” runs validation phases in the `consumer` container.

### 3.2 Containers

Both defined in [`repro/docker-compose.yml`](repro/docker-compose.yml):

- **toolchain-builder** ([`docker/toolchain-builder.Dockerfile`](repro/docker/toolchain-builder.Dockerfile)) â€” Ubuntu 22.04 with build tooling. `/work/build` and `/work/src` are on named Docker volumes (`gcc-win98-build`, `gcc-win98-src`) â€” keeps heavy I/O off the Windows host's 9p layer. Wine + Xvfb live here too for in-build smoke checks. Note: Ubuntu's apt meson (0.61) is too old for muon; we `pip3 install meson` on top, lands at `/usr/local/bin` ahead of `/usr/bin`.
- **consumer** ([`docker/consumer.Dockerfile`](repro/docker/consumer.Dockerfile)) â€” multi-stage; the `extractor` stage unpacks all three tarballs into `/opt/cross-toolset`, `/opt/native-toolset`, `/opt/extras`. Ships Wine + Xvfb for smoke tests.

### 3.3 Build phases (canonical order)

1. **Cross** (Linuxâ†’Win98): fetch â†’ patch â†’ cross-binutils â†’ cross-mingw-w64 â†’ cross-gcc-stage1 â†’ cross-pthread9x â†’ cross-gcc-final â†’ verify â†’ **win98-compat shim** â†’ **install-pe-checker** â†’ package â†’ manifest. (The win98-compat shim installs `libwin98compat.a` + `win98_compat.h` into the cross sysroot at `out/toolchain/i686-w64-mingw32/{lib,include}` so the package step rolls it into the cross tarball. install-pe-checker drops a standalone copy of [`pe-win98-check.sh`](repro/scripts/verifiers/pe-win98-check.sh) + the allowlist/denylist JSONs under `out/toolchain/share/win98-verify/` with a `bin/pe-win98-check` wrapper symlink, so downstream users of the cross tarball get the same Win98 PE sanity-check tool we use during the build.)
2. **Native** (Canadian Cross, Win98â†’Win98): build-native-mingw-deps (gmp/mpfr/mpc for the host) â†’ build-native-mingw-w64 â†’ build-native-host-gcc â†’ build-native-binutils â†’ build-native-pthread9x â†’ verify â†’ **install-win98-compat-native** â†’ **install-win98-helpers-native** â†’ **strip-native-toolset** â†’ package â†’ manifest. (install-win98-compat-native file-copies `libwin98compat.a` + `win98_compat.h` from the cross sysroot into the native sysroot at `out/native-toolset/i686-w64-mingw32/{lib,include}/` â€” same target triple, bit-identical .a â€” and bundles the source under `out/native-toolset/share/win98-compat/{src,include}/` for reference. Lets on-Win98 ports use `-include win98_compat.h` / `-lwin98compat` the same way the extras tools do. install-win98-helpers-native drops [`setenv.bat`](repro/scripts/win98/setenv.bat) + [`check-versions.bat`](repro/scripts/win98/check-versions.bat) into the toolset root so the zipped package ships them next to `bin/`. setenv.bat fixes the Win98-SE-default-unset HOME / HOMEDRIVE / HOMEPATH / TMP / TEMP env vars that gdb, busybox sh/vi, make, ctags, and muon all care about; check-versions.bat is a `--version` smoke-test sweep for every shipped tool. strip-native-toolset walks every `.exe`/`.dll` and strips debug sections â€” autoconf defaults bake in `-g -O2` so cc1.exe ships ~299 MB unstripped vs ~27 MB stripped; dedupes by inode so GCC's hardlink families get stripped once.)
3. **Extras** (Win98-hosted user tools â€” gated on `BUILD_EXTRAS=1`): busybox â†’ ctags â†’ make â†’ diffutils â†’ patch â†’ gdb â†’ muon â†’ jq â†’ bcrypt-shim â†’ bb-shims â†’ consdiag â†’ sockdiag â†’ verify â†’ **install-win98-helpers-extras** â†’ **install-pe-checker-extras** â†’ **strip-extras-toolset** â†’ write-extras-build-info â†’ package â†’ manifest. Order is cheapest-first so failures surface early; the bcrypt shim sits between muon and verify because the shim install goes into `out/extras-toolset/bin/` and must be present before verify scans. install-win98-helpers-extras drops the same [`setenv.bat`](repro/scripts/win98/setenv.bat) + [`check-versions.bat`](repro/scripts/win98/check-versions.bat) as the native phase; shipping in both packages removes a "which one do I install first" footgun. install-pe-checker-extras mirrors install-pe-checker but targets `out/extras-toolset/share/win98-verify/` â€” same script content (single source of truth), so Win98 users can run `sh share\win98-verify\pe-win98-check.sh foo.exe` against any binary on the box (jq.exe from extras + objdump.exe from native must be on PATH). The script itself is POSIX-sh and runs in busybox-w32 ash, not just bash; see [`docs/plans/pe-check-posix-rewrite.md`](repro/docs/plans/pe-check-posix-rewrite.md) for the rewrite history. strip-extras-toolset mirrors the native-side step (~140 MB â†’ ~7 MB on gdb.exe alone) and MUST come before write-extras-build-info, which records bin/ SHAs into BUILD.TXT.
4. **Smoke**: layout check, PE check on native + extras binaries, CMake+Ninja builds of `tests/` under both toolchains, all run/checked under Wine.

### 3.4 Conventions

- **Naming**: `build-cross-*.sh` vs `build-native-*.sh`. New native tools (cross/native or extras) should be `build-native-<tool>.sh`.
- **Step sentinels**: every script ends with `mark_done <step-name>`; later scripts gate on `require_step <step-name>`. Files live at `out/.status-<STATUS_SCOPE>-<step>` where `STATUS_SCOPE` defaults to `${TARGET}__m${MATRIX}` (prevents resume from skipping into a different config). Names must match between scripts and the `*_STEPS` arrays.
- **Common library**: source [`lib/common.sh`](repro/scripts/lib/common.sh) at the top of every script. It provides `log`, `die`, `run_logged`, `mark_done`, `is_done`, `skip_if_done`, `invalidate_if_stale`, `require_step`, `require_dir`, `require_file`, `require_executable`, `builder_exec`, `builder_script`, `fetch_component`, and the dir vars `ROOT_DIR`, `SRC_DIR`, `BUILD_DIR`, `LOG_DIR`, `OUT_DIR`, `PATCH_DIR`, `PREFIX`, `TARGET`, `JOBS`. Plus two Win98-flag pairs you propagate into every native build's configure/Make: `WIN98_TARGET_CPPFLAGS`/`WIN98_TARGET_LDFLAGS` (header gates + PE DllCharacteristics â€” always required) and `WIN98_COMPAT_CPPFLAGS`/`WIN98_COMPAT_LDFLAGS` (force-include the win98-compat shim header + link `-lwin98compat` â€” required for any binary whose source references a function in the Â§5.6 coverage list).
- **Auto-invalidate stale install/build sentinels**: handled in TWO layers â€” the runner is primary; the in-script helper is a backstop for direct invocations.
  - **Runner layer (primary, what `./build.sh` uses)**: applies to BOTH the toolchain runner ([`run-toolchain-build.sh`](repro/scripts/run-toolchain-build.sh) â€” `CROSS_STEPS`/`NATIVE_STEPS`/`EXTRAS_STEPS`) and the smoke runner ([`run-smoke-pipeline.sh`](repro/scripts/run-smoke-pipeline.sh) â€” `SMOKE_STEPS`). Each step entry carries an optional trailing inputs field â€” colon-separated paths relative to the repo root, after the `env` field for toolchain steps or after `description` for smoke steps. The helper `invalidate_step_if_stale` (in [`lib/common.sh`](repro/scripts/lib/common.sh)) checks those plus the step script itself; if any is newer than the sentinel, the sentinel is removed before the `is_done_in_builder` / `is_done` check, so the step re-runs. Sentinels live under `/work/out` which is bind-mounted, so the host can `stat` them directly â€” no docker exec needed for the staleness check. This is the layer that matters when iterating via `./build.sh`. For smoke steps, the natural input is the corresponding package manifest (`out/package/gcc-win98-{cross,native,extras}-toolchain[-extras].json`) â€” its mtime updates whenever the toolset is repackaged, so the smoke test re-runs exactly when there's a new artifact to validate.
  - **In-script layer (backstop for direct invocation)**: `invalidate_if_stale <step> [<input>...]` does the same thing but inside the step script. Used to be the primary mechanism, but for runner invocations the runner short-circuits via `is_done_in_builder` BEFORE running the script â€” so the in-script helper never fired. The runner-layer fix made it primary; the in-script call is fine to keep (and worth keeping when you invoke a step directly via `docker compose exec ... build-foo.sh` for iteration) but no longer load-bearing in the standard `./build.sh` flow.
  - **When to add inputs to a step definition**: any step whose effective inputs are stable in-tree files we edit directly â€” install scripts, in-tree shim builds (bb-shim, bcrypt-shim, win98-compat, sockdiag, consdiag), build-native-busybox (config edits), install-pe-checker (verifier script + allowlists). Intentionally NOT applied to `build-*` steps that pull from `/work/src/` (upstream source trees â€” those follow a different cadence driven by `fetch-sources.sh` and config.json refs).
  - **Shared-library-output exception**: `build-*` steps that link `-lwin98compat` (currently `build-native-binutils` plus all the EXTRAS `build-native-*` consumers) DO declare `out/toolchain/i686-w64-mingw32/lib/libwin98compat.a` as an input even though they pull source from `/work/src/`. The intent isn't to track source changes â€” it's to chain through the shim *output* so a shim edit re-fires every consumer that links it. The same input is repeated on the downstream verify/strip/package/manifest steps so the whole tail re-runs in lockstep; otherwise the freshly-relinked binaries get silently skipped at packaging and you ship a half-updated toolset. **Repeat it on the CROSS-side `package` + `write-toolchain-manifest-v2` too** â€” the cross tarball is what the consumer image's `COPY` reads to populate `/opt/cross-toolset/`, and the smoke-cmake-build step links against that copy of `libwin98compat.a`. Without the cross-side input, a shim rebuild updates `/work/out/toolchain/...libwin98compat.a` but the cross tarball stays stale, docker layer-cache reuses the old COPY layer, and smoke binaries link against the previous shim. (How this was caught: `_fstat64` wrapper added to the shim source, build-win98-compat re-ran and the .a in /work/out got the new symbols, but smoke binaries still imported `msvcrt:_fstat64` because the consumer's `/opt/cross-toolset/...libwin98compat.a` was the pre-`_fstat64` copy.) Generalizes to any future shared static library we build in-tree.
  - **Postmortem context**: this two-layer design exists because the in-script helper alone had a silent regression â€” the runner's pre-skip caused `invalidate_if_stale` to never run on `./build.sh` invocations. The canonical instance was the install-pe-checker staleness bug; the busybox-config-edit-not-rebuilding bug surfaced the same root cause and motivated lifting invalidation to the runner. If you add a new step with editable in-tree inputs, prefer the runner-layer mechanism.
- **Logging**: `run_logged <log-name>.log <cmd...>` tees to `logs/`. Every step has its own log.
- **Sources fetched via [`config.json`](repro/config.json)**: pinned commits (and SHA-checksummed tarballs where available). Add new components by extending the matrix entry, updating the `MAPPING` in [`config_matrix_exports.py`](repro/scripts/lib/config_matrix_exports.py), adding `*_FETCH_*`/`*_TARBALL_*`/`*_COMPONENT_VERSION` placeholders to [`common.sh`](repro/scripts/lib/common.sh), and calling `fetch_component` in [`fetch-sources.sh`](repro/scripts/fetch-sources.sh). Also add to the `retry-clean.sh` sweep list.
- **Prefer release tarballs over git** when both are available. `fetch_component` takes both â€” if `tarball.url` is set in config.json, it's used and the git source is metadata only. Tarballs are faster to extract (no per-file checkout) and ship pre-generated `configure` so we sidestep `./bootstrap` / `autoreconf` dependency drift.
- **Canadian Cross compiler hygiene**: when building native (Win98-hosted) tools, prepend `$REPO_ROOT/out/toolchain/bin` to `PATH` and pass `--host=i686-w64-mingw32`. The simpler `build-native-*` scripts (make, diffutils, ctags) are good templates.
- **Packaging**: the cross tarball is made with `XZ_OPT=-1 tar -cJf` (low compression â€” large file, trades disk for time). The native + extras zips are made with `zip -9 -r` from a `cp -al`-staged tree under the renamed top-level (`gcc_win98/` or `gcc_win98_extras/`); the staging step is what lets us rename the prefix without zip's missing `--transform` equivalent. The stage dir MUST be on the same filesystem as the source (we use `mktemp -d -p "$PACKAGE_DIR" ...`, NOT the default `/tmp`) â€” `cp -al` fails with `EXDEV` across devices, and in the toolchain-builder container `/work` is a bind-mount while `/tmp` is the overlay FS. All three archives land in `out/package/`.

### 3.5 Patch system

- [`repro/patches/`](repro/patches/) holds per-component patch series. `apply-patches.sh` applies `series.txt` in order with `git apply`.
- New patched components need: `patches/<name>/<version>/series.txt` + `*.patch` files, **plus** a `patch.py` `PatchSet` subclass if you want regeneration support via `generate-patches.py`. Hand-rolled patches without the generator class work fine â€” generator support is a nice-to-have.
- For one-off mingw header gaps in a single component, an inline build-script CPPFLAGS or stub-include approach (see Â§5 patch / muon) is usually less overhead than a full patch series.

## 5. Porting playbook â€” patterns and gotchas

These came out of porting the extras tools. Pattern-match these against new tools before reinventing.

### 5.1 mingw header gaps

mingw-w64 lacks POSIX symbols Windows has no analog for. The cheapest fix for each:

- **`SA_RESTART`, `SIGHUP`, `SIGPIPE`, `SIGSTOP`, `SIGQUIT`, `SIG_BLOCK`, `SIG_UNBLOCK`, `SIG_SETMASK`**: stuff defaults via CPPFLAGS (`-DSIGHUP=1 -DSIGPIPE=13` etc â€” pick unused-on-Windows signal numbers). At runtime `signal()` returns `SIG_ERR` for these, the handler doesn't install, and since the signal can't fire on Windows it's a no-op.
- **`<sys/resource.h>`**: ship a stub header on the include path. See `build-native-patch.sh` for the pattern â€” a build-script heredoc writes `mingw-stubs/sys/resource.h` with typedef `rlim_t`, `struct rlimit`, and macros for `getrlimit`/`setrlimit` that return success with placeholder limits.
- **`getuid`, `getgid`, `geteuid`, `getegid`**: macro them to `(0)` via a force-included compat header â€” Windows has no POSIX users; everything runs as if root. Combined with the resource stub, force-include via `-include $BUILD_DIR/mingw-stubs/compat-mingw.h`.
- **`<regex.h>` (POSIX C regex)**: mingw has no POSIX regex (only C++ `<regex>`). Either link gnulib-regex, link libgnurx, or drop the tool. cscope was dropped for this plus curses.

### 5.2 Multi-package builds (binutils-gdb, gcc)

The top-level binutils-gdb `configure` passes `--disable-option-checking` to sub-configures. This silently swallows unknown flags â€” you'll think your `-Dfoo=bar` is taking effect when it isn't. Always check the sub-configure's `--help` for the exact flag name. Caught this with gdb 11's `--with-libgmp-prefix` vs the misleading-looking `--with-gmp` (the latter is a top-level mingw-deps flag, not gdb's).

The gdb in the `binutils-2_36_1` tree is **gdb 11.0.50-dev**, not gdb 10.1 â€” gdb's release cadence is independent of binutils.

### 5.3 Symbol collisions

Linking against pthread9x AND mingw-w64's `libmsvcrt.a` can collide on symbols both provide as Windows-old-CRT fallbacks. We hit this with `strtoull` (pthread9x's `int64.c` and mingw's `lib32_libmsvcrt_extra` both supply it). Fix: `-Wl,--allow-multiple-definition` in `LDFLAGS`. The two impls are interchangeable; first-on-the-link-line wins.

Pure-C extras tools didn't trip this because they don't pull in `-lpthread`. C++ tools (gdb) do because libstdc++ drags it in for thread support.

### 5.4 Cross-compile with meson

For meson-based projects, the bootstrap-the-build-system path is often a dead end (e.g. muon's bootstrap CLI is too minimal for cross-files). Just use Python meson + ninja from the builder:

1. Write a `mingw32.cross.ini` with `[binaries]` pointing at `$CROSS_BIN_DIR/${TARGET}-*` and `[host_machine]` declaring `system='windows', cpu='i686'`.
2. Set `[properties]` `needs_exe_wrapper = true` so meson doesn't try to run cross-built executables on the host. If the project needs to run produced helpers mid-build, add `exe_wrapper = 'wine'` instead.
3. `meson setup --cross-file=... build-dir source-dir`; `ninja -C build-dir`; manually copy artifacts to install dir (meson's install path will try to invoke the cross-built binaries).

### 5.5 Configure quirks worth knowing

- `--disable-dependency-tracking` cuts a lot of useless work in cross-builds â€” every autoconf-based extras script uses it.
- `MAKEINFO=true` skips info-page generation (we ship none); pass it to `make` for autoconf tools that try to rebuild texinfo docs.
- `--disable-gcc-warnings` on gnulib-using projects (diffutils, patch) saves time and noise â€” gnulib's `-Werror` defaults are aggressive.

### 5.6 Compile-time API shim (win98-compat)

For *missing functions* on *present DLLs* â€” the case where Win98's `KERNEL32.DLL` (or `ADVAPI32`, `WS2_32`, `MSVCRT`) exists but doesn't export some entry point the compiler bound as a static PE import â€” there's a project-wide static-archive shim at [`repro/win98-compat/`](repro/win98-compat/). It installs `libwin98compat.a` + `<win98_compat.h>` into the cross-toolchain sysroot. Consumers pick it up via `WIN98_COMPAT_LDFLAGS="-Wl,--whole-archive -lwin98compat -Wl,--no-whole-archive"` from [`lib/common.sh`](repro/scripts/lib/common.sh); `WIN98_COMPAT_CPPFLAGS` is **empty** by default â€” the shim no longer force-includes its header. Because the install location is the sysroot, the cross-toolchain tarball ships it for downstream porting too.

Each wrapper does `GetProcAddress` against the real system DLL at runtime: full behavior on NT-class hosts, behavior-preserving fallback on Win9x (sensible failure return / synthesized data â€” see [`repro/win98-compat/src/win98_compat.c`](repro/win98-compat/src/win98_compat.c) for what each fallback does).

**Three layered hook techniques (all in [`win98_compat.c`](repro/win98-compat/src/win98_compat.c)).** When a consumer's source-level call to e.g. `getaddrinfo(...)` reaches the linker, our shim catches it via one of three mechanisms â€” picked by what the compiler emitted in the .o file, not by anything the build script does differently. They share the same `win98_FOO` wrappers; the difference is purely how the linker is steered to them. `--whole-archive` is required regardless of technique: it forces every symbol below into the link at the point of `-lwin98compat`, beating the system import libs to definition no matter where the autotools rule positions LDFLAGS.

1. **IAT slot interception (primary, no source/header involvement).** Most callers see the API via `windows.h`/`ws2tcpip.h`/etc. with `__declspec(dllimport)` on the declaration, so gcc emits an indirect `call *__imp__FOO@N`. The bottom of `win98_compat.c` has an inline-asm block that defines `__imp__FOO@N` symbols in `.rdata` as 4-byte pointers to `_win98_FOO@N`. Linker resolves the consumer's reference against our slot first (forced in by `--whole-archive`); no IAT entry for `FOO` from the real DLL gets emitted; the indirect call at runtime lands directly in our wrapper. This is the mechanism that fires for almost every shipped binary â€” it's why `WIN98_COMPAT_CPPFLAGS` can be empty.

2. **Direct-call thunks (.text jmp shims).** Some code hand-declares the API without `dllimport` (e.g. busybox's libbb/yescrypt declares `SystemFunction036` itself), so gcc emits a direct `call _FOO@N` with no IAT indirection. The same asm block defines `_FOO@N` `.text` thunks that `jmp _win98_FOO@N`. Without these thunks, the linker would satisfy `_FOO@N` from advapi32.dll's short import-library member â€” and that .o ALSO defines `__imp__FOO@N` in `.idata$5`, colliding with our IAT slot. So the thunk doesn't just route the call; it keeps the system import lib's symbol-providing .o out of the link entirely.

3. **Source-level macro redirect (header, opt-in).** [`win98_compat.h`](repro/win98-compat/include/win98_compat.h) still ships in the sysroot and still does `#define FOO win98_FOO` (after preloading the system headers in `winsock2.h` â†’ `ws2tcpip.h` â†’ `windows.h` order, then forward-declaring the `win98_*` wrappers without `dllimport`). Source-header-preload-then-redefine is what keeps `dllimport` off the wrapper's own declaration â€” without it, the linker hunts `_imp__win98_FOO` and fails to link (the v1 bug). This path is no longer used by our own build (the empty `WIN98_COMPAT_CPPFLAGS` means no `-include` happens automatically), but it's the published interface for downstream consumers of the cross tarball who want to call `win98_*` explicitly without --whole-archive plumbing.

Current coverage:

- **kernel32**: GetFinalPathNameByHandleA, GetSystemWow64DirectoryA, GetLogicalProcessorInformation, GetSystemTimePreciseAsFileTime, IsWow64Process, GetProcessId, GetConsoleWindow, GetFileSizeEx.
- **ws2_32**: getaddrinfo, freeaddrinfo, getnameinfo â€” these are now **real `gethostbyname`-based emulations** on Win9x (not just `EAI_FAIL` stubs); single IPv4 A-record, AI_PASSIVE/AI_NUMERICHOST/AI_NUMERICSERV honored, AI_CANONNAME ignored. Lazy-resolves the underlying Winsock-1.1 primitives via `GetProcAddress` so consumers that link `libwin98compat` but never call getaddrinfo don't grow new ws2_32 imports.
- **advapi32**: SystemFunction036 (RtlGenRandom).
- **msvcrt**: qsort_s, `_fstat64` (wraps `_fstati64` and widens the three time fields from `__time32_t` â†’ `__time64_t`; Win98 SE's msvcrt has `_fstat`/`_fstati64` but not the XP-era `_fstat64`. Needed because mingw-w64's static stdio init pulls `_fstat64` into every `-static` binary that touches stdio â€” was silently failing the smoke PE check until pe-check itself was tightened in commit 05e8d24), `_mkgmtime32` / `_mkgmtime64` (pure-C UTC-tm-to-time_t. Win98 SE's msvcrt has bare `mktime`/`gmtime`/`localtime` but not the Microsoft-extension `_mkgmtime`; mingw's libmsvcrt thunks both `_mkgmtime32` and `_mkgmtime64` C-source names back to the bare `_mkgmtime` import in the DLL, so we intercept both suffixes â€” needed by jq's `my_mktime` `#elif WIN32` branch when timegm and tm_gmtoff are both absent, will fire for any future timezone-aware consumer too).

For a per-function audit of who-calls-each-shimmed-symbol (cosmetic vs load-bearing, fixed vs unfixed), see [`repro/docs/win9x-shim-audit.md`](repro/docs/win9x-shim-audit.md). Maintain it when you add or remove a shimmed function.

When to add a function:

1. **Confirm** it's a real binding in a shipped binary via `objdump -p`. Vista+ symbols the compiler tries-and-dead-strips don't need shimming.
2. **Add** a `#define` in [`win98_compat.h`](repro/win98-compat/include/win98_compat.h) and a probe-then-fallback impl in [`win98_compat.c`](repro/win98-compat/src/win98_compat.c). Add it to the `EXPECTED_SYMBOLS` spot-check in [`build-win98-compat.sh`](repro/scripts/build-win98-compat.sh) if it's a new DLL bucket.
3. **Don't** add a function whose fallback would have to misrepresent its API (returning fake results that break callers downstream). Drop the consumer or patch out the specific call site instead.

When NOT to use this pattern (use a bundled-DLL drop-in Â§5.7 instead):

- The whole DLL is missing on Win98 (`bcrypt.dll`, `libcurl.dll`, etc.). A static archive can't satisfy the loader's DLL-existence check.
- The consumer dynamically `LoadLibrary`s a missing DLL by name. Same reason.

### 5.7 Bundled-DLL shims for unresolvable static imports

When the compiler bakes a static PE import of a DLL that doesn't exist on Win98 â€” and the import is too deep in a third-party library to patch out â€” ship a tiny shim DLL with the same name + exports next to the consuming `.exe`. Win98's App Directory search resolves DLLs by name from the binary's own directory before walking system paths, so a same-name shim wins.

The bcrypt shim is the canonical example: libstdc++ 11's `std::random_device` statically imports `bcrypt!BCryptGenRandom`, that dependency travels into anything linked `-static-libstdc++` (here, `gdb.exe`), and Win98 has no `bcrypt.dll`. The shim is ~10 lines of C ([`repro/bcrypt-shim/bcrypt.c`](repro/bcrypt-shim/bcrypt.c)) implementing only the function gdb actually imports, delegating to msvcrt's `rand` seeded from `GetTickCount` (non-cryptographic â€” acceptable because gdb only uses the entropy as a seed source, not for keys).

The recipe:

1. **Confirm scope** with `objdump -p <exe> | awk '/DLL Name: <shim>.dll/,/DLL Name:/'` â€” list every function the consumer actually imports. Only stub those; resist adding plausible-looking siblings.
2. **Cross-build with `--kill-at`** so stdcall exports come out undecorated (libstdc++ imports `BCryptGenRandom`, not `BCryptGenRandom@16`). Also pass `WIN98_TARGET_CPPFLAGS` / `WIN98_TARGET_LDFLAGS` so the shim itself is Win98-safe.
3. **Self-verify** the shim DLL through `pe_check_win98` in its build script before installing â€” catches a shim that accidentally pulls in a Vista+ API of its own.
4. **Install to `out/<toolset>/bin/`** alongside the consumer, NOT to a separate directory. Win98's App Directory search needs them adjacent.
5. **Declare to the verifier**: `export PE_CHECK_BUNDLED_DLLS="<shim>.dll"` before the verify scan. The verifier still applies the full PE check to the shim itself; it just lets the consumer's import resolve. Set this in *every* verifier that scans the toolset (currently [`verify-extras-package.sh`](repro/scripts/verifiers/verify-extras-package.sh) and [`smoke-check-extras-pe.sh`](repro/scripts/smoke-check-extras-pe.sh)).
6. **Wire as its own EXTRAS_STEP** between the consumer's build and the package-verify step. Don't fold it into the consumer's build script â€” the shim should be independently re-buildable.

If you find yourself wanting to add a second function to an existing shim, stop and ask whether the underlying tool needs to be there at all. The shim should stay ~10 lines; growth signals that the actual dependency is heavier than this pattern was designed for, and a libstdc++ / source patch may be cleaner.

### 5.8 Vista+ APIs in third-party source

For worked-example walk-throughs of past Win9x investigations (the round-by-round breakdown of how a symptom narrowed to a root cause), see [`repro/docs/win98-debug-history.md`](repro/docs/win98-debug-history.md).

The Win98 `_WIN32_WINNT=0x0400` setting hides any mingw-w64 declaration gated behind a higher target. Third-party projects that feature-detect at compile time (rather than via configure-time probes) hit this as `unknown type name` / `implicit declaration` errors on Vista-era APIs like `FILE_NAME_INFO`, `GetFileInformationByHandleEx`, `BCryptGenRandom`, `RtlGenRandom`.

Pick by who's calling and why:

- **The caller is dead code on Win98** (e.g. muon's MSYS/mintty pty detection â€” Cygwin/MSYS can't run on Win98) â†’ inline-patch the source from the build script to wrap the block in `#if _WIN32_WINNT >= 0x0600 ... #endif`. Make the patch idempotent (check for an inserted marker comment before applying). See `build-native-muon.sh` for the pattern â€” a python heredoc with brace-matching.
- **The caller is live and the missing symbol is a function on a DLL Win98 does have** (e.g. `kernel32!GetFinalPathNameByHandleA`) â†’ use the compile-time API shim (Â§5.6). Add the function to its coverage and you're done â€” no per-tool patch.
- **The caller is live and the missing symbol is the whole DLL** (e.g. `bcrypt.dll`, `libcurl.dll`) â†’ bundled-DLL drop-in (Â§5.7).
- **The caller is live and there's no honest fallback** (crypto primitives the tool genuinely uses, etc.) â†’ disable the codepath via configure, drop the tool, or fork the source. Don't ship a fake.
- **The caller is in mingw-w64's own CRT** (e.g. `libmingwex.a::ftruncate64.o` pulling in `FindFirstVolumeW` for a disk-space check). That's a source-level patch against our mingw-w64 source tree â€” see `repro/patches/mingw-w64/master/0002-ftruncate64-skip-volume-walk-pre-win2k.patch` for an example that gates the offending block on `_WIN32_WINNT >= 0x0500`.
- **The caller is in a Win9x kernel API gap that isn't just "DLL/symbol missing"** (e.g. `WaitForMultipleObjects` is exported and resolves, but rejects console input handles as wait objects â€” an NT-4+ feature). The allowlist/denylist won't catch it (the export is real, the call returns `WAIT_FAILED` at runtime), so the consumer's runtime logic has to detect Win9x and take a polling fallback. The example is `repro/patches/binutils-gdb/2.36.1/0001-mingw-hdep-poll-console-input-on-win9x.patch`: it adds a `gdb_select_is_win9x()` startup probe (`GetVersionEx` â†’ `VER_PLATFORM_WIN32_WINDOWS`) and a `PeekConsoleInput`-based polling path in `gdb/mingw-hdep.c` so gdb's event loop doesn't spin printing `select: No Error.` on Win98. Pattern when you hit a similar trap in another tool: lazy-cache the Win9x detection in a static, leave the NT-path untouched, and write the Win9x path against the lowest-common-denominator Win9x APIs (`PeekConsoleInput`, `WaitForSingleObject(timeout=0)`, `Sleep`).
  - Three more flavors of the same Win9x-API-misbehaves trap, all surfaced by the busybox-w32 patch series:
    - **`SetConsoleMode` silently accepts unknown flag bits on Win9x.** busybox-w32's `terminal_mode()` auto-detect tries `ENABLE_VIRTUAL_TERMINAL_PROCESSING` (a Win10+ flag) and demotes to Console API emulation if `SetConsoleMode` rejects it. Win9x doesn't validate the flag â€” it stores the bit and reports success â€” so the demotion never fires, mode stays at `CONFIG_TERMINAL_MODE=5`, and `ansi_emulate` passes ESC sequences straight through. On a `command.com` console with no ANSI.SYS, every cursor move (like backspace's `ESC[1D`) prints literally. Fixed in `repro/patches/busybox-w32/master/0005-terminal-mode-force-console-api-on-win9x.patch`: detect Win9x at startup and force `mode=0`, bypassing the SetConsoleMode probe.
    - **Unnamed `CreateFileMapping` HANDLEs aren't shareable across `CreateProcess` on Win9x.** busybox-w32's ash uses a "forkshell via shared memory" trick to simulate fork(): create an unnamed file mapping (`CreateFileMapping(INVALID_HANDLE_VALUE, ..., NULL)`) with `bInheritHandle=TRUE`, sprintf the raw HANDLE value into argv, spawn busybox-as-sh again with `--fs <hex-handle>`, child does `MapViewOfFile((HANDLE)strtoul(argv[2], ...))`. Works on NT because handle inheritance preserves the numeric value; doesn't work on Win9x because HANDLE values are per-process. The spawn itself succeeds, but the child's `MapViewOfFile` fails â€” or rather, on Win9x the failure surfaces upstream as `_spawnve == -1` and the user gets `sh: unable to spawn shell` the moment they type any command. Fixed in `0004-forkshell-named-mapping.patch`: use a *named* mapping (`busybox-fs-<pid>-<counter>`), pass the name through argv instead of the handle, child does `OpenFileMapping(name)`. Works on both NT and Win9x; `bInheritHandle=TRUE` still keeps the kernel object alive across the parent's `CloseHandle` via cross-process refcount.
    - **Win9x `CreateProcess` rejects forward slashes in `lpApplicationName`.** busybox-w32 caches `bb_busybox_exec_path` from `GetModuleFileName(NULL, ...)` and then runs it through `bs_to_slash()` (cosmetic, for POSIX-shaped error messages). All the `mingw_spawn_*` paths feed that forward-slashed string straight into `_spawnve` â†’ `CreateProcess`. NT silently normalizes; Win9x doesn't, the spawn fails, and the same "unable to spawn shell" surfaces. Fixed in `0003-spawn-backslash-path.patch`: in `spawnveq`, dup the path and `slash_to_bs()` it before calling `spawnve`. The fix is always-on, not Win9x-gated â€” the conversion is safe on NT and saves a fragile platform check.
  - General intuition for these: the Win9x kernel is *permissive in the wrong ways*. Unknown flag bits to `SetConsoleMode` succeed silently. Forward slashes to `CreateProcess` fail silently. Inherited handle slots renumber silently. Modern Windows is uniformly stricter or uniformly permissive, so testing only on NT/Wine misses every one of these. Wine in particular emulates NT behavior closely â€” none of the three busybox patches above can be validated under Wine, only on a real Win9x system. Plan for "the issue ships even though smoke tests pass" and write the fix to be as conservative as possible (force a known-good mode, use named-not-handle, normalize-don't-trust-input).

### 5.9 PE verifier `set -e` interaction

`pe_check_win98` returns 1 on a Win98-incompatible binary as a normal status code, not an error â€” the caller reads `$PE_CHECK_RESULT` for the verdict and `$PE_CHECK_FAIL_REASON` for the why. But the verifier scripts run under `set -e`, which treats rc=1 as fatal and kills the loop before the `case` runs â€” masking the actual failure with a silent exit 1. Every caller MUST wrap the call: `pe_check_win98 "$f" || true`. There are currently five callers, all fixed; mirror the pattern in any new verifier.

Same trap, different cause: awk `exit` inside a `printf '%s\n' "$dump" | awk ...` pipeline closes the pipe early. Under `set -o pipefail` the upstream `printf` gets `SIGPIPE` and the assignment fails with rc=141, killing the whole verify. Don't use `exit` inside awk pipelines on small inputs â€” let awk read to EOF.

Third trap, very different cause: the per-function check parses `objdump -p` output, and the column count of that output **varies by binutils version**. There are two layouts in the wild:

- **3-col** (`vma  Hint/Ord  Member-Name  Bound-To`) â€” what the toolchain-builder container's Ubuntu 22.04 binutils 2.38 emits. Example row: `\t999fe\t 1373  GetTokenInformation`.
- **4-col** (`vma  Ordinal  Hint  Member-Name  Bound-To`) â€” what newer binutils emit (e.g. mingw-w64 host objdump). Example row: `\t0009952c  <none>  055d  GetTokenInformation`.

A regex anchored on the wrong count silently matches zero rows â€” the verifier reports green while the per-function check is effectively a no-op. [`pe-win98-check.sh`](repro/scripts/verifiers/pe-win98-check.sh) now tries the 4-col regex first and falls back to the 3-col regex, so it auto-detects which layout is in front of it â€” but bumping the base image again could introduce a third layout (5-col Bound-To populated, etc.). If you bump binutils or notice the function-allowlist check passing on a binary you *know* has a forbidden import, generate a fresh `objdump -p` against the active toolchain and confirm at least one of the two patterns still matches a real import row.

Heads-up for cross-environment debugging: running `objdump -p` from the **host** (mingw64 binutils) and from the **toolchain-builder container** (Ubuntu binutils 2.38) on the same `.exe` produces *different layouts*. If you reproduce a verifier issue interactively, run inside the container (`docker compose exec -T toolchain-builder bash -c '...'`) to match what CI sees â€” otherwise you may convince yourself the regex is broken when it's actually only broken for one side. Both regex layouts are exercised in [`pe-win98-check.sh`](repro/scripts/verifiers/pe-win98-check.sh)'s import-row block and either one matching is sufficient.

(History: this trap has shipped a regression *in both directions* â€” once when the regex was anchored to a phantom 4th column that never appeared in container output, and once when it was tightened from 4-col back to 3-col after a host-environment debugging session falsely concluded the 4-col layout was wrong. Auto-detect ends the ping-pong.)

### 5.10 FAT32 applet shims (busybox)

On POSIX, busybox is installed as one `busybox` binary plus a per-applet symlink for each enabled tool (`ls` â†’ `busybox`, `cp` â†’ `busybox`, ...). busybox dispatches on `argv[0]`. FAT32 has no symlinks, no hardlinks â€” neither feature has ever been part of the FAT spec â€” so on Win98 we can't use that mechanism. Three options that *don't* work, for the record:

- **Full copies of `busybox.exe` per applet** (what `busybox --install` without `-s` does). 700 KB Ă— ~50 applets = 35+ MB, dominated by waste.
- **`.bat` wrappers** (`cp.bat = @busybox cp %*`). Fine from `command.com`, but inside busybox sh the `execvp("cp", ...)` path goes through `mingw_execvp` which has only fragile support for `.bat` dispatch (it shells out to `cmd.exe`, with the usual `.bat` quoting horrors). Fails for scripted use.
- **`.sh` wrappers** (`exec busybox cp "$@"`). Works inside `sh` but not from `command.com` â€” the user typing `cp foo bar` at `C:\>` gets "Bad command or file name."

What works: a small `bb-shim.exe`, copy-renamed once per applet (~112 applet copies currently). At [`repro/bb-shim/bb-shim.c`](repro/bb-shim/bb-shim.c) â€” ~80 lines, builds to ~49 KB statically linked. Each shim does at runtime:

1. `GetModuleFileNameA(NULL, ...)` to learn its own path (e.g. `C:\OPT\EXTRAS\BIN\LS.EXE`).
2. Strip the directory and `.exe` to get the applet name (`ls`).
3. Locate `busybox.exe` in the same directory.
4. `_spawnv(_P_WAIT, busybox_path, {"busybox", "ls", original-args...})` and exit with the child's exit code.

This is the standard busybox dispatch trick, just split out into its own tiny binary. Works from `command.com` AND inside `busybox sh` (it's a real `.exe`, both shells take the normal `CreateProcess` path). Exit code propagates because `_P_WAIT` waits. The shim is built once and `cp`-installed under each curated applet name by [`build-bb-shims.sh`](repro/scripts/build-bb-shims.sh) â€” see the `SHIM_APPLETS` array there for the current list (compression, text encoding, math/util, process/timing, ed, checksums â€” POSIX basics only, no networking applets shimmed; `nc` and `httpd` are runnable from `busybox sh` but not exposed as top-level `.exe`s, and `wget`/`whois`/`ftp*` are disabled in the busybox config â€” see [`repro/docs/networking-on-win98.md`](repro/docs/networking-on-win98.md)).

Why `_spawnv` not `_execv`: msvcrt's `_execv` documents itself as "the calling process is terminated", but on Windows there's no true `execve` â€” internally it does CreateProcess-then-exit. The child runs concurrently with the parent's exit, which means `cmd.exe`'s prompt can race ahead of the actual applet's output. `_spawnv P_WAIT` is the synchronous-spawn-and-wait that POSIX users expect from `cp` etc. The shim stays resident (~46 KB) for the duration of the child's execution; this is the only real cost.

Win9x case-coercion gotcha: `GetModuleFileNameA` returns the path the loader used to launch the .exe, and `command.com` on Win98 hands back the **FAT 8.3 SHORT name in uppercase** even when the file's LFN is mixed-case ("ls.exe" on disk â†’ `C:\OPT\EXTRAS\BIN\LS.EXE` as seen by the running process). busybox's applet table is case-sensitive and all entries are lowercase, so the shim lowercases its derived applet name (`_strlwr`) before dispatching. Wine preserves case, so this bug doesn't reproduce under Wine â€” only on real Win98 â€” which is exactly the class of mistake Â§5.8's intuition warns about. Any future shim/wrapper that derives behavior from `GetModuleFileName` or `argv[0]` on Win98 needs the same normalization.

When to add an applet to `SHIM_APPLETS`:

1. **Confirm it's enabled** in the busybox config: `grep CONFIG_<NAME>=y repro/configs/busybox-w32.config`. The shim has no idea what busybox compiled in, so a shim for a disabled applet would print "applet not found" at runtime.
2. **Don't shim a name we ship as a standalone**: `sh.exe` (busybox copy), `cmp.exe`/`diff.exe` (diffutils), `patch.exe`, `make.exe`, `ctags.exe`, `gdb.exe`, `muon.exe`, `consdiag.exe`, `sockdiag.exe`, `bcrypt.dll`, `bb-shim.exe` (the master copy). The install loop in [`build-bb-shims.sh`](repro/scripts/build-bb-shims.sh) refuses to overwrite a binary that isn't already a shim copy, but the right move is to keep the SHIM_APPLETS list disjoint from the standalone-tools list in the first place.
3. **Skim the applet's `_main` for Linux assumptions** if it's something you haven't seen run on Windows. Most file/text/process applets work; networking applets (`ifconfig`, `route`, `ping`) and process-control applets (`mount`, `umount`) generally don't.

The shim itself is Win98-clean (msvcrt + KERNEL32 only) and goes through `pe_check_win98` in its build script. The per-applet copies are byte-identical to `bb-shim.exe`, so once the master copy passes the verifier, the copies all pass too â€” but `verify-extras-package.sh` still scans every `.exe` in `bin/`, which surfaces a regression in the shim immediately on the next build instead of hiding behind a "we already verified the master" optimization.

### 5.11 tinycc â€” second compiler on Win98

tcc is a Win98-hosted second C compiler that ships alongside `gcc.exe` in the extras toolset. It's not a replacement â€” gcc remains the toolchain â€” but it's small (~380 KB stripped) and useful for "compile and run this little C file on Win98 without firing up gcc" workflows. The host binary is built from the upstream mob branch via a Linuxâ†’Win98 cross-compile, and the target-side runtime library (`libtcc1.a`) is built by running tcc.exe itself under wine. Three configure-stage traps you have to know about; the install layout is unusual; the runtime risk surface is shallow.

**Configure trap 1 â€” ccache mangling via `--cross-prefix`.** tcc's configure applies the cross prefix by *string concatenation* on `$CC`, not by tokenization (`configure`:298, `cc="${cross_prefix}${cc}"`). Our [`common.sh` ccache integration](repro/scripts/lib/common.sh) auto-sets `CC="ccache i686-w64-mingw32-gcc"` for every `build-native-*.sh`, so adding `--cross-prefix=i686-w64-mingw32-` produces the nonsense command `i686-w64-mingw32-ccache i686-w64-mingw32-gcc`. Fix: `unset CC CXX` after sourcing common.sh and pass `--cc=${TARGET}-gcc --ar=${TARGET}-ar` directly. tcc's `build_cross=yes` branch still triggers from `--cpu=i386` vs build host `x86_64`, so dropping `--cross-prefix` doesn't lose anything. The same trap will bite any other tool with a configure that does string-prepend cross-prefix logic; check before reaching for `--cross-prefix` in a new build script.

**Configure trap 2 â€” `c2str.exe` must be disabled via `--config-predefs=no`.** tcc has a build-time helper (`conftest.c -DC2STR` â†’ `c2str.exe`) that converts `include/tccdefs.h` into an embeddable C string literal at `tccdefs_.h`. The Makefile rule for it is `$S./c2str.exe $< $@` â€” but cross-built, c2str.exe is a Win32 PE that won't exec on the Linux build host (the shell tries to interpret the PE bytes as a script and prints diagnostic gibberish). `--config-predefs=no` empties the `TCCDEFS_H` variable so nothing depends on `tccdefs_.h`, and `tccpp.c:3625-3629` falls back to `#include <tccdefs.h>` at runtime â€” the file is loaded from `dirname(tcc.exe)/include/tccdefs.h`, which our install step ships. This is also what upstream's `build-tcc.bat` does (the `c2str.exe` invocation is commented out there for the same reason). The lesson generalizes: when a cross-build wants to *run* a freshly-cross-compiled host helper, either feature-disable the helper or wine-wrap the invocation â€” there's no third option, the kernel won't run the PE.

**Configure trap 3 â€” `libtcc1.a` needs wine.** Same problem as trap 2 but inside the libtcc1.a sub-make: `lib/Makefile` compiles each `lib/*.c` (and `lib/*.S`) by invoking `$(XCC) -c <src>` where XCC defaults to `$(XTCC)` = `../tcc.exe`. We wrap with a tiny `wine-tcc.sh` script and pass `make libtcc1.a XTCC="$WINE_TCC"`. The single override covers compile AND archive steps because `XAR = $(XTCC) -ar` (tcc-as-archiver). Built single-job because the lib sub-make includes the top Makefile and the parallel-build hazard around `c2str.exe`/`tccdefs_.h` would otherwise need a tcc-specific serialization (see upstream Makefile comment at line 274-279) â€” single-job here costs only ~20 small compiles.

**Install layout â€” sibling `tcc/` tree via a runtime tccdir patch.** On Win32, tcc resolves its `tccdir` (where it looks for `include/`, `lib/`, `libtcc1.a`) at runtime as `dirname(tcc.exe)` ([`libtcc.c::config_tccdir_w32`](https://github.com/TinyCC/tinycc/blob/mob/libtcc.c)). With `tcc.exe` in the shared `bin/` next to the other extras tools, the unpatched runtime would look for resources at `bin/include`, `bin/lib`, etc. â€” polluting the bin namespace. [`patches/tinycc/mob/0001-config-tccdir-relative-to-bin.patch`](repro/patches/tinycc/mob/0001-config-tccdir-relative-to-bin.patch) augments the function to append `/../tcc` to its dirname result, so the lookup lands at `<prefix>/tcc/` instead. Final layout: `bin/tcc.exe`, `tcc/include/`, `tcc/lib/{libtcc1.a,*.def}`, `tcc/libtcc/{libtcc.a,libtcc.h}` â€” a clean Unix-ish split with no launcher binary, no `.bat` wrapper, and no second PATH entry for the user. The patch is 8 bytes of `strcat` plus a comment block; install path collapse (`bin/../tcc` â†’ `tcc`) is handled downstream by `CreateFile`/`fopen`. Only fires when `CONFIG_TCCDIR` isn't pre-baked, which is our case for `--targetos=WIN32 --prefix=<dir>` builds (configure block at `configure:413-419` sets `tccdir_auto=yes` and skips the `print_str CONFIG_TCCDIR` step).

**Runtime risk surface â€” small in practice.** Host-side audit: tcc.exe touches only KERNEL32 basics (`GetModuleFileNameA`, `VirtualAlloc/Free`, `FreeLibrary`), msvcrt (`_fullpath`, `_snprintf`, `_strtoi64`), and `SetUnhandledExceptionFilter` (Win95+). The one Vista+ API call site (`AddVectoredExceptionHandler` at `tccrun.c:1430`) sits under `#ifdef _WIN64` and is dead code on our i386 target. [`tcc.h:55-56`](https://github.com/TinyCC/tinycc/blob/mob/tcc.h) defensively hardcodes `_WIN32_WINNT 0x502`; we override to `0x0400` via `WIN98_TARGET_CPPFLAGS` and the upstream's WIN64-gated dead-strip handles the rest. Target-side audit (binaries tcc PRODUCES): `tccpe.c` writes `OS=4.0` headers naturally, so produced PEs are Win98-clean by default â€” the [Phase 3d/3e smoke tests](repro/scripts/smoke-tcc-build-c.sh) confirm this for a curated set of `tests/smoke-c/` programs covering libtcc1.a soft-float, libm (via msvcrt), stdio, file I/O, and time. **Not yet exercised**: pthread (tcc bundles a minimal-mingw `include/` set, which doesn't ship `pthread.h` and tcc's `.def` library list omits pthread9x â€” adding both is the phase-3 follow-up), winsock (same `.def` plumbing concern), SEH, and C++ (tcc is C-only by design).

**When you add an option or change configure flags, run the smoke pipeline.** Two phases gate tcc-related changes: `smoke-tcc-build` (single hello world â€” PE-check + wine-run) and `smoke-tcc-build-c` (the curated subset). Both live as inputs of `EXTRAS_MANIFEST` plus their respective `.c` files, so a build-script edit auto-invalidates them and resume re-fires. The smoke-c subset is the right place to ADD new test cases when a fix lands â€” keep deferring the pthread/winsock/SEH/etc. set with their reasons baked into the script's header until the underlying plumbing exists.

## 6. Things to avoid

- **Don't introduce UCRT anywhere.** Any new component's configure must select msvcrt (`--with-default-msvcrt=msvcrt` for mingw-w64 sub-builds; for tools, link against `-lmsvcrt` if it matters).
- **Don't enable threads/features that require Vista+ APIs.** When in doubt, the PE verifier will catch it â€” but it's a slow feedback loop.
- **Don't use the Ubuntu `gcc-mingw-w64-i686-win32` package as the final compiler for anything shipped.** It's GCC 10 and uses Vista+ thread-atexit. Always prepend `out/toolchain/bin` to `PATH`.
- **Don't lose reproducibility.** New components need pinned commits in `config.json`; don't fetch tarballs with floating URLs. When pinning a git source after a successful build, capture the resolved SHA with `git -C /work/src/<name> rev-parse HEAD` and paste it into the `commit` field.
- **Don't skip the PE check.** Every shipped `.exe`/`.dll` must pass `pe_check_win98`. The extras verifier ([`verifiers/verify-extras-package.sh`](repro/scripts/verifiers/verify-extras-package.sh)) handles this for the extras package; mirror the pattern for new families.
- **Don't `--no-verify`, `--force`, `--amend`, or `rm -rf` anything load-bearing without asking.** The build is long and resumable â€” preserve in-flight state.
- **Don't ship a meson project without `--buildtype=release`.** Meson defaults to `buildtype=debug` (-O0 -g), which is invisible until something prompts a binary-size audit. Autoconf projects bake in `-g -O2` by default so they're fine on auto-pilot, but meson is the exception. Caught when muon.exe was the only shipped binary running unoptimized â€” see the muon row in Â§4.

## 7. Useful runtime facts

- **Build environment**: Windows 11 host (PowerShell + Git Bash), but the build itself runs in Linux containers via Docker Compose. Use the Bash tool for repo work â€” scripts under `repro/` are bash and the dev loop is `docker compose exec`.
- **Resumability**: status sentinels under `out/.status-*` make the pipeline resumable. `./build.sh` (or `./scripts/run-toolchain-build.sh --resume`) picks up where it left off. To force a step to re-run, delete its sentinel: `rm /work/out/.status-*-<step-name>`. To force re-fetch of one tool's source, also delete `src/<tool>/.tarball-extracted` (for tarball components) or `rm -rf src/<tool>` (for git components).
- **Resume vs sentinel invalidation**: `--resume` skips steps before the resume point regardless of sentinel state â€” so if you need an earlier step to re-run, *don't* use `--resume`; just nuke its sentinel and let the runner walk the full list (it'll skip done-steps via `is_done_in_builder` and run the un-done ones).
- **MSYS path conversion** (Git Bash on Windows): `docker compose exec` rewrites `/work/...` to `C:/git-sdk-64/work/...` before passing to the container. `in_container_path()` in common.sh prepends a leading `/` (so `//work/...`) which MSYS leaves alone; in-container bash collapses double-slash per POSIX. Every `builder_*` / `consumer_*` helper goes through this.
- **Status reporters** live in [`scripts/utils/`](repro/scripts/utils/) â€” `project-status.sh` aggregates the lot, including the extras phase via `extras-toolset-status.sh`.
- **Logs** are in [`repro/logs/`](repro/logs/), timestamped per step.
- **Paths from PowerShell** are `u:\home\brian\github\gcc-for-Windows98\...`. Bash tool sees them as `u:/home/brian/github/gcc-for-Windows98/...`.
